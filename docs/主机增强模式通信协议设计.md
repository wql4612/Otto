# 主机增强模式通信协议设计

> 日期：2026-06-24  
> 对应计划：`docs/plan.md` 阶段 2  
> 适用范围：当前 `AI_Desktop_Assistant` 固件，以及后续新增的主机端语音服务

## 一、先说明这份设计的依据

这份协议不是照搬外部样例，而是根据当前仓库实际状态整理出来的。

已确认的现状如下：

1. ESP32 端已经具备轻量 HTTP API：
   - `GET /api/status`
   - `POST /api/action?cmd=...`
   - 由 [`wifi_driver.cpp`](/home/thd/CS/Project/Otto/AI_Desktop_Assistant/main/wifi_driver.cpp) 提供
2. ESP32 端仍保留了一套手写 WebSocket 服务能力：
   - 可以收文本消息
   - 可以主动广播文本
   - 后续可扩展为主机通信通道
3. 板端已经有可复用的结构化动作执行层：
   - `apply_qwen_voice_json(...)`
   - `show_face_by_name(...)`
   - `servo_motion_from_name(...)`
   - 位于 [`main.ino`](/home/thd/CS/Project/Otto/AI_Desktop_Assistant/main/main.ino)
4. 板端当前语音链路是：
   - 本地唤醒
   - 板端录音
   - 云端 ASR / Planner
   - 返回 JSON
   - 板端执行动作
5. 当前仓库里还没有现成可运行的 `server/` 主机服务目录，因此这一步要先把协议定清楚，再按协议补代码。

---

## 二、设计目标

主机增强模式不是推翻当前固件，而是在保留独立运行能力的前提下，把最重的语音链路迁移到主机。

目标分为三条：

1. `Standalone Mode` 继续可用，作为无主机时的备用方案。
2. `Host-Enhanced Mode` 下，主机优先接管流式 ASR、对话规划、TTS。
3. 两种模式共用同一套“动作执行协议”，避免后续重复维护两套表情/舵机/灯光逻辑。

---

## 三、推荐的总体结构

### 3.1 模式划分

#### Standalone Mode

由 ESP32 自己完成：

- 本地唤醒词检测
- 录制语音
- 调用 Qwen / ASR
- 解析动作 JSON
- 控制屏幕、舵机、灯光、提示音

#### Host-Enhanced Mode

由主机接管重负载部分：

- 流式 ASR
- 对话理解
- 动作规划
- TTS 生成

ESP32 保留：

- 麦克风采集
- 扬声器播放
- 屏幕表情显示
- 舵机执行
- 超声波 / 摄像头 / 其他基础传感器
- 基础状态机

### 3.2 为什么不直接把所有服务都放回 ESP32

因为当前板端已经同时承担：

- 本地唤醒推理
- 摄像头抓拍
- 云端请求构造
- 屏幕刷新
- 舵机动作
- 轻量 HTTP 服务

继续把流式 ASR、流式 TTS 和完整会话编排也压在板端，演示效果会明显受限。主机增强模式的意义，就是把“最影响体验的那部分”迁移出去。

## 四、通信方式选择

结合当前仓库代码，推荐采用下面的混合通信方案。

### 4.1 保留 HTTP API

HTTP API 继续保留，用于：

- 调试页轮询状态
- 人工触发测试动作
- 非实时配置读写

这部分已经存在，改动最小。

### 4.2 将现有 WebSocket 服务升级为“主机控制通道”

推荐让主机主动连接 ESP32 的 WebSocket 服务，原因有三点：

1. 当前固件已经有 WebSocket 服务端实现，可直接扩展。
2. 主机主动连板子，符合你前面提出的“由主机主动获取信息”的思路。
3. 这样不需要 ESP32 额外引入新的长连接客户端栈，新增工作量更可控。

因此推荐结构为：

- ESP32：保留轻量 HTTP server + WebSocket server
- 主机：作为 WebSocket client 主动连到 ESP32

### 4.3 语音数据与控制消息分离

为了后续实现稳定，建议在同一条 WebSocket 逻辑链路上区分两类数据：

1. 文本 JSON 控制消息
2. 二进制 PCM 音频帧

规则如下：

- 文本帧：用于握手、状态切换、动作 JSON、TTS 控制
- 二进制帧：用于板端上传麦克风 PCM 音频

这样做的好处是：

- 协议清晰
- 主机端好解析
- 后续如果改成独立的音频通道，也不影响动作协议本身

## 五、推荐的协议分层

### 5.1 板端固定保留的接口

#### HTTP 层

- `GET /api/status`
- `POST /api/action?cmd=...`

用途：

- 网页调试
- 主机读设备状态
- 主机做非实时配置

#### WebSocket 层

建议专门作为“主机增强模式通道”使用。

用途：

- 主机声明接管语音
- 板端上传音频流
- 主机下发表情/舵机/灯光动作
- 主机下发 TTS 控制

### 5.2 后续新增的主机端职责

主机端服务建议拆为四块：

1. `session_manager`
   - 管理与 ESP32 的连接
   - 管理模式切换
2. `asr_pipeline`
   - 接收 PCM 流
   - 送云端或本地 ASR
3. `planner`
   - 根据文本返回统一动作 JSON
4. `tts_pipeline`
   - 生成语音
   - 通知 ESP32 播放

## 六、消息协议定义

下面的协议是“最小可落地版本”，优先保证阶段 3 能形成闭环。

### 6.1 通用字段

所有 JSON 文本消息建议都带：

```json
{
  "type": "message_type",
  "ts": 1761270000,
  "session_id": "optional-session-id"
}
```

字段说明：

- `type`：消息类型
- `ts`：毫秒级时间戳或本地递增计数都可
- `session_id`：当前语音会话标识；无会话时可省略

### 6.2 主机连接握手

#### ESP32 -> Host: `hello`

在 WebSocket 建立后，ESP32 主动发送：

```json
{
  "type": "hello",
  "device_id": "otto-esp32s3",
  "fw": "v3",
  "mode": "standalone",
  "capabilities": {
    "local_wake": true,
    "pcm_uplink": true,
    "screen": true,
    "servo180": true,
    "speaker_local": true,
    "tts_stream_pull": false
  }
}
```

说明：

- `tts_stream_pull` 当前先标为 `false`
- 因为现阶段板端还没有真正完成“从主机持续拉取 TTS 音频流并播放”的实现

#### Host -> ESP32: `host_claim`

主机若要接管语音链路，则发送：

```json
{
  "type": "host_claim",
  "protocol_version": 1,
  "voice_mode": "stream",
  "planner_mode": "host",
  "tts_mode": "host"
}
```

ESP32 收到后切换到 `Host-Enhanced Mode`，并暂停板端云端语音主链路。

#### Host -> ESP32: `host_release`

主机主动释放接管：

```json
{
  "type": "host_release",
  "reason": "shutdown"
}
```

ESP32 收到后回退到 `Standalone Mode`。

### 6.3 录音阶段控制

#### ESP32 -> Host: `wake`

板端本地模型检测到唤醒词后发送：

```json
{
  "type": "wake",
  "session_id": "wake-0001",
  "source": "local_wake",
  "energy": 1234
}
```

用途：

- 通知主机进入一轮新的语音会话
- 主机据此清空上轮状态

#### Host -> ESP32: `capture_start`

```json
{
  "type": "capture_start",
  "session_id": "wake-0001",
  "sample_rate": 16000,
  "format": "pcm_s16le",
  "channels": 1,
  "frame_ms": 20,
  "max_duration_ms": 5000
}
```

ESP32 收到后：

- 停止板端云语音流程
- 进入“专注本轮语音”的状态
- 必要时暂停会抢资源的任务
- 开始向主机发送 PCM 帧

#### ESP32 -> Host: `capture_end`

```json
{
  "type": "capture_end",
  "session_id": "wake-0001",
  "reason": "timeout"
}
```

表示这轮录音已结束。

### 6.4 二进制音频帧

在 `capture_start` 之后，ESP32 通过 WebSocket 二进制帧连续发送原始音频：

- 编码：`PCM S16LE`
- 采样率：`16000`
- 声道：`1`
- 帧长：建议 `20ms`

主机以“收到 `capture_start` 后的二进制帧都属于该 `session_id`”进行归属，不需要在每个二进制包内重复包 JSON 头。

如果后续发现定位帧边界或丢包恢复不够方便，再升级成：

- `4B magic`
- `2B seq`
- `2B payload_len`
- `payload`

但第一版不必一开始就做复杂。

### 6.5 主机返回文本与动作

#### Host -> ESP32: `action`

这是最核心的统一动作协议，建议直接兼容当前板端已有的 JSON 执行格式：

```json
{
  "type": "action",
  "session_id": "wake-0001",
  "result": "succeed",
  "face": "happy",
  "servo": "happy",
  "light": "on",
  "answer": "欢迎回来，我已经帮你把灯打开了。"
}
```

字段含义：

- `result`：`succeed` / `error`
- `face`：`idle` / `happy` / `confused` / `surprised` / `listening`
- `servo`：与当前 `servo_motion_from_name(...)` 对齐
- `light`：`on` / `off` / `keep`
- `answer`：后续交给 TTS 或页面展示

如果失败：

```json
{
  "type": "action",
  "session_id": "wake-0001",
  "result": "error",
  "face": "confused",
  "servo": "confused",
  "light": "keep",
  "answer": "抱歉，我没有听清。"
}
```

### 6.6 TTS 控制协议

TTS 建议分两阶段实施。

#### 第一阶段：只下发文字，不做主机语音回放

此时：

- 主机只发送 `action.answer`
- ESP32 仅执行表情和舵机动作
- 需要声音反馈时仍可先播放板端本地短音效

这是最稳的最小闭环。

#### 第二阶段：主机接入 TTS 播放

建议新增三条控制消息：

##### Host -> ESP32: `tts_start`

```json
{
  "type": "tts_start",
  "session_id": "wake-0001",
  "audio_format": "wav",
  "sample_rate": 16000
}
```

##### Host -> ESP32: `tts_url`

```json
{
  "type": "tts_url",
  "session_id": "wake-0001",
  "url": "http://192.168.1.10:9000/tts/current.wav"
}
```

##### Host -> ESP32: `tts_stop`

```json
{
  "type": "tts_stop",
  "session_id": "wake-0001"
}
```

推荐原因：

- 文字控制仍走 WebSocket
- 实际音频数据由 ESP32 用 HTTP 拉取
- 不必一开始就在 WebSocket 内同时混发上行 PCM 和下行大块音频

这和你前面找到的样例架构也更接近，但这里保留了当前项目自己的执行层。

### 6.7 中断与重置

#### Host -> ESP32: `stop`

```json
{
  "type": "stop",
  "session_id": "wake-0001",
  "reason": "interrupt"
}
```

ESP32 收到后：

- 停止当前录音
- 停止当前播放
- 恢复到可再次唤醒的状态

#### Host -> ESP32: `reset`

```json
{
  "type": "reset",
  "reason": "hotword_interrupt"
}
```

用于硬重置当前会话链路。

#### ESP32 -> Host: `state`

建议板端主动回报关键状态变化：

```json
{
  "type": "state",
  "mode": "host_enhanced",
  "voice_stage": "idle",
  "wake_listener": true,
  "speaker_busy": false,
  "person_present": true
}
```

## 七、板端状态机建议

当前板端已经有：

- 本地唤醒监听
- 二阶段语音命令处理
- 视觉状态机

主机增强模式建议只在语音链路上额外加入模式分支。

### 7.1 语音状态建议

```text
IDLE
  -> WAKE_DETECTED
  -> ACK_PLAYING
  -> HOST_CAPTURING / STANDALONE_RECORDING
  -> HOST_WAIT_ACTION / STANDALONE_WAIT_QWEN
  -> HOST_TTS_PLAYING (可选)
  -> RESUME_IDLE
```

### 7.2 切换原则

#### 当 `mode = standalone`

- 走当前已有的 `run_cloud_voice_command(...)`

#### 当 `mode = host_enhanced`

- 不再由板端自己发 ASR / Planner 请求
- 改为：
  - 本地 wake
  - 向主机发 `wake`
  - 等待 `capture_start`
  - 上传 PCM
  - 等待 `action`
  - 执行动作

### 7.3 与其他模块的协调

当进入第二阶段语音会话后，建议像你之前提出的一样，尽量暂停会抢资源的任务：

- 暂停人脸探测触发
- 暂停高频调试日志
- 暂停非必要动画
- 若后续 TTS 播放和麦克风共用 I2S / 音频资源，则播放阶段禁止录音

下面这些位置后续可以直接作为实现入口。

### 8.1 模式枚举

建议在 [`main.ino`](/home/thd/CS/Project/Otto/AI_Desktop_Assistant/main/main.ino) 新增：

```cpp
enum RuntimeMode {
    MODE_STANDALONE = 0,
    MODE_HOST_ENHANCED = 1
```

并维护：

- 主机连接状态
- 当前会话 `session_id`

### 8.2 动作执行层复用

当前已有的：

- `apply_qwen_voice_json(...)`
- `show_face_by_name(...)`
- `servo_motion_from_name(...)`

建议抽成更中性的名字，例如：

- `apply_action_json(...)`

这样无论动作来自：

- 板端 Qwen

都能走同一执行层。

### 8.3 WebSocket 回调扩展

当前已经存在：

- `wifi_set_ws_message_handler(...)`
- `on_ws_message(...)`

后续可以把旧的调试消息与新的主机协议分开：

- `type = cmd`：保留给旧调试

### 8.4 麦克风上传入口

当前板端已有麦克风采集能力，但还没有“连续上行 PCM 到主机”的环节。

建议新增：

- `voice_start_host_capture(session_id)`
- `voice_stop_host_capture()`
- `ws_send_binary_pcm(...)`

第一版先只做“单轮会话上传”，不要一开始就做永久流式上传。

## 九、分阶段落地建议

### 阶段 2A：先完成协议与模式切换骨架

目标：

- 不接主机 ASR
- 先让 ESP32 能识别“主机已接管”

要做的事：

1. 新增 `RuntimeMode`
2. 新增 `host_claim / host_release`
3. 把当前板端语音主链路包进模式判断

验收：

- 主机连上后，ESP32 进入 `MODE_HOST_ENHANCED`
- 主机断开后，自动回退到 `MODE_STANDALONE`

### 阶段 2B：实现板端上行 PCM

目标：

- 能把一次唤醒后的录音发到主机

要做的事：

1. wake 后不直接调 Qwen
2. 改为向主机发 `wake`
3. 主机回 `capture_start`
4. 板端连续发 PCM 二进制帧

验收：

- 主机能收到完整一轮音频

### 阶段 2C：实现主机动作回传

目标：

- 主机把 ASR / Planner 的结果回给板端

要做的事：

1. 主机发送 `action`
2. 板端复用当前 JSON 执行层

验收：

- 主机一句话可驱动表情、舵机、灯光

### 阶段 2D：实现主机 TTS

目标：

- 主机生成语音，板端播放

要做的事：

1. 先做 `tts_start + tts_url + tts_stop`
2. 再决定要不要升级为真正的流式 PCM 下发

验收：

- 板端可播放主机生成的回复音频

## 十、当前版本的明确边界

这份设计里，下面这些点是刻意保守的：

1. 不要求一开始就删掉板端 HTTP server
2. 不要求一开始就做全双工音频
3. 不要求一开始就做连续对话
4. 不要求一开始就让主机接管视觉链路

当前最优先的是：

- 先把主机增强模式的语音闭环接通
- 保住独立模式作为备用
- 让最终展示具备“双模式”的工程说服力

---

## 十一、一句话结论

当前最合适的方案不是“把现有系统推倒重来”，而是：

> 保留 ESP32 现有轻量 HTTP 服务和本地执行层，把现有 WebSocket 通道升级为主机增强模式的语音控制链路，并统一使用结构化动作 JSON 作为板端执行入口。

这样做改动最小，也最符合当前仓库实际状态。
