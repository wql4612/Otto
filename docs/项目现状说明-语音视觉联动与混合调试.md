# 项目现状说明：语音视觉联动与混合调试

> 日期：2026-06-24  
> 适用范围：当前 `AI_Desktop_Assistant` 主固件版本  
> 目的：说明最近一轮功能接入后的系统现状、使用方式与后续待完成部分

---

## 一、当前阶段目标

本轮工作的重点不是最终产品形态，而是先把几条核心功能链路接通：

1. 语音链路：本地唤醒 + 云端识别
2. 视觉链路：距离检测触发 + 云端人脸存在判断
3. 调试链路：从板端重网页托管切换为“电脑端页面 + ESP32 轻量 API”的混合方案
4. 硬件资源整理：暂时移除 RF 控制，优先保证语音、视觉、屏幕与基础执行器稳定工作

---

## 二、语音识别链路现状

### 2.1 已完成部分

当前语音部分已经接入为“两阶段识别”结构：

- 第一阶段：**本地模型持续监听唤醒词**
- 第二阶段：唤醒成功后，**录制一段语音并发送到阿里云 / Qwen 侧做识别**

目前这条链路已经可以完成以下流程：

1. 本地监听到唤醒词
2. 播放 `wake.wav` 作为反馈
3. 进入云端指令录制阶段
4. 发送音频到云端 ASR / Qwen
5. 解析简单控制结果
6. 执行当前已接通的基础动作，例如开灯、关灯

### 2.2 当前定位

这部分目前仍然偏“功能验证”：

- 本地模型负责低成本、持续运行的唤醒
- 云端模型负责更高质量的语音识别与后续扩展

### 2.3 尚未完成部分

虽然语音链路已经打通，但后续还需要补足更完整的交互联动：

- 云端识别结果与**屏幕表情**联动
- 云端识别结果与**舵机动作**联动
- 更复杂的“情感沟通”或多轮对话反馈
- 更细的指令结构化解析，而不仅仅是基础开关类测试

换句话说，当前已经具备“能听到并能执行”的基础能力，但还没有完全进入“情感表达 + 外设联动”的完整机器人交互阶段。

---

## 三、人脸识别与屏幕控制链路现状

### 3.1 已完成部分

视觉部分目前不再采用“摄像头持续工作、持续上传”的模式，而改成了：

- **超声波测距**先做近距离目标检测
- 只有在距离满足条件时，才触发摄像头拍照一次
- 将照片发送给云端进行**是否有人脸**的判断
- 再根据结果决定屏幕亮灭

### 3.2 当前行为逻辑

当前链路可概括为：

1. 超声波每隔一段时间测距
2. 当检测到近距离物体进入阈值范围
3. 摄像头拍摄一帧
4. 云端判断画面中是否存在人脸
5. 若检测到人脸：屏幕点亮并显示表情
6. 若未检测到人脸：屏幕保持黑屏

### 3.3 这样设计的意义

相比最早的“摄像头常开 + 连续上传”，现在的好处是：

- 降低摄像头持续运行负担
- 降低网络请求频率
- 更符合独立设备长期运行的资源控制需求
- 让“屏幕唤醒”更接近真实在场感知逻辑

### 3.4 当前限制

现阶段视觉侧仍以“是否有人脸存在”为主，不做身份识别，也不做表情/情绪等更细粒度分析。

---

## 四、网页调试方式的变化

### 4.1 架构调整

网页调试部分已经从“ESP32 自己托管完整页面”切换为**混合策略**：

- **电脑端**负责托管调试页面
- **ESP32** 只保留轻量 HTTP API

对应目录：

- 电脑端调试页：`desktop_debug_panel/`
- ESP32 侧接口：`wifi_driver.cpp/h`

### 4.2 当前工作方式

当前使用流程为：

1. ESP32 连接局域网
2. 电脑在本地启动静态文件服务
3. 浏览器打开电脑端调试页
4. 调试页通过 HTTP 调用 ESP32 的 `/api/status` 与 `/api/action`

### 4.3 这样调整的原因

这么做的主要目的是：

- 减少 ESP32 托管网页带来的资源占用
- 避免开发板长期承担前端页面服务
- 保留轻量调试能力
- 方便后续让开发板更多时间留给语音、视觉和硬件任务

### 4.4 当前调试页定位

网页端现在更偏向于：

- 查看日志
- 查看实时状态
- 手动测试单个硬件功能
- 测试语音 / 测距 / 屏幕 / 舵机等链路

而不是最终用户产品界面。

---

## 五、RF 模块现状

RF 控制在当前阶段已**暂时移除和停用**。

原因很明确：

- 当前优先级不在 RF 联动
- 板上引脚与资源更需要优先保证语音、屏幕、摄像头、超声波和舵机工作
- 继续保留 RF 只会增加调试复杂度

因此现阶段的处理原则是：

- **不初始化 RF**
- **不把 RF 作为当前主链路的一部分**
- **网页调试页中移除 RF 相关测试入口**

如果后续重新接回 RF，应在硬件资源再次核对后单独恢复。

---

## 六、当前系统的整体理解

可以把当前项目理解为一个“已打通核心主链路，但仍在补完整交互”的版本：

- 语音：已经能从“本地唤醒”走到“云端识别并执行”
- 视觉：已经能从“距离触发”走到“拍照识别人脸并控制屏幕”
- 调试：已经能通过电脑端页面完成主要状态观测与功能测试
- 硬件资源：正在逐步整理，优先保障关键模块稳定

---

## 七、后续建议方向

按照当前结构，下一阶段更适合继续推进以下内容：

1. 完善语音识别结果与屏幕表情、舵机动作之间的联动
2. 将“云端识别出的语义”转成更稳定的结构化动作
3. 完善视觉侧的人脸判断后行为，例如欢迎、倾听、待机等表情切换
4. 在当前混合调试架构稳定后，再决定哪些模块值得重新接回（如 RF）

---

## 八、一句话总结

当前版本已经完成了：

- **本地唤醒 + 云端语音识别**
- **超声波触发 + 云端人脸判断 + 屏幕亮灭**
- **电脑端调试页 + ESP32 轻量 API 的混合调试方案**
- **RF 暂时移除，优先保证核心功能稳定**

但“情感交互联动”和“更多外设动作联动”仍然属于下一阶段要继续完善的内容。
