# AI Desktop Assistant 双模式演示计划书

> 日期：2026-06-24  
> 适用范围：`AI_Desktop_Assistant` 当前主固件与后续主机增强方案  
> 目的：明确项目后续不偏题的实施路线，形成“独立运行 + 主机增强”双模式架构，并为最终 Demo 汇报做准备

---

## 一、计划目标

本阶段不再继续把所有能力都压在 ESP32 端，而是明确建立两种运行模式：

1. `Standalone Mode`：开发板独立运行
2. `Host-Enhanced Mode`：开发板连接主机，主机承担语音重负载能力

最终希望在展示时形成这样的项目叙事：

- 设备本身具备独立工作能力
- 连接主机后可获得最佳交互体验
- 两种模式共享同一套硬件系统与大部分外设执行逻辑

---

## 二、当前现状判断

### 2.1 已具备的能力

当前板端已经具备以下基础：

- 本地唤醒词监听
- 云端语音识别与动作执行
- 超声波触发 + 摄像头抓拍 + 云端人脸存在判断
- 屏幕表情显示
- 180 度舵机控制
- 扬声器播放本地反馈音
- 电脑端轻量调试面板

### 2.2 当前主要问题

虽然功能链路已基本打通，但 ESP32 当前承担的任务偏多：

- 本地唤醒推理
- 音频录制
- 云端请求构造与解析
- 摄像头抓拍与上传
- 传感器轮询
- 屏幕刷新
- 舵机执行
- 板端 HTTP 服务与调试接口

这会导致：

- 实时性不足
- 多模块之间更容易争抢资源
- 语音流式体验难以做到自然
- 演示时效果上限受限于开发板性能

---

## 三、总体架构方向

后续架构采用“双模式共享执行层”的思路。

### 3.1 Standalone Mode

定位：无主机连接时的独立运行方案。

特点：

- ESP32 自己完成基础感知与执行
- 保留本地唤醒
- 保留当前云端图片识别 / 云端语音识别简化链路
- 允许性能上有所折中

目标：

- 能脱离主机独立完成基础互动
- 可作为备用模式与答辩中的“离线/半离线能力证明”

### 3.2 Host-Enhanced Mode

定位：有主机参与时的最佳体验方案。

特点：

- 主机接管流式 ASR、对话规划与 TTS 合成
- ESP32 退化为“音频前端 + 外设执行终端”
- 降低 ESP32 运算与编排压力

目标：

- 提升语音互动自然度
- 支持更稳定的连续对话与语音播报
- 作为最终展示时的主用模式

---

## 四、两种模式的职责边界

### 4.1 ESP32 固定职责

无论是否连接主机，ESP32 都保留以下职责：

- 本地传感器接入
- 屏幕表情显示
- 舵机动作执行
- 本地音频采集
- 本地音频播放
- 基础状态机
- 与电脑端调试页的 HTTP 状态接口

### 4.2 仅在 Standalone Mode 中由 ESP32 承担的职责

- 本地唤醒监听
- 板端发起云端语音识别请求
- 板端发起云端对话/动作 JSON 请求
- 板端未来可能发起简化 TTS 请求

### 4.3 仅在 Host-Enhanced Mode 中由主机承担的职责

- 流式 ASR
- 对话编排
- 情绪 / 表情 / 外设动作规划
- TTS 音频生成
- 会话打断控制
- 多阶段音频链路状态管理

---

## 五、后续语音系统目标形态

### 5.1 Standalone Mode 语音链路

建议保留为：

1. 本地唤醒词检测
2. 板端录制固定长度音频
3. 云端 ASR
4. 云端返回标准 JSON
5. 板端解析 JSON
6. 控制表情、舵机、灯光、音频反馈

这是“低依赖、可独立运行”的版本。

### 5.2 Host-Enhanced Mode 语音链路

建议升级为：

1. ESP32 连续采集 PCM 音频
2. 发送到主机服务端
3. 主机做流式 ASR
4. 主机做 AI 对话规划
5. 主机做 TTS 音频生成
6. 主机将文字结果与音频结果返回
7. ESP32 播放语音、切换表情、执行舵机动作

这是“高效果展示”的版本。

---

## 六、模式切换原则

后续模式切换不应依赖手工修改大量代码，而应由状态机统一管理。

建议采用以下原则：

1. 开机默认进入 `Standalone Mode`
2. 若检测到主机服务可达，则切换到 `Host-Enhanced Mode`
3. 若主机失联，则自动回退到 `Standalone Mode`

切换时应保证：

- 麦克风录音链路状态一致
- 扬声器播放状态可重置
- 当前语音会话可以被终止并重新开始
- 屏幕与舵机的执行层不感知“上层是谁发的命令”

---

## 七、统一动作协议目标

为了让 Standalone 与 Host-Enhanced 共用一套执行层，后续所有高层语义结果都应统一为结构化动作协议。

当前建议的最小协议：

```json
{
  "result": "succeed",
  "face": "happy",
  "servo": "happy",
  "light": "on",
  "answer": "欢迎回来，我已经帮你把灯打开了。"
}
```

后续可扩展字段：

- `audio_mode`
- `speaker`
- `screen_text`
- `servo_180_angle`
- `servo_180_pattern`
- `interrupt`
- `follow_up`

原则：

- ESP32 不直接理解复杂自然语言
- ESP32 只执行结构化动作
- 主机模式和板端模式尽量共享同一执行协议

---

## 八、分阶段实施计划

### 阶段 1：稳定当前板端方案

目标：

- 让 Standalone Mode 成为可靠备用方案

任务：

- 继续稳定本地唤醒
- 稳定当前云端 JSON 动作链路
- 保证表情、舵机、灯光联动可重复演示
- 继续控制日志量，避免调试功能反向拖慢主链路

验收标准：

- 独立上电时可完成“唤醒 → 识别 → 执行动作”
- 人脸与超声波链路稳定
- 网页调试不再影响核心运行

### 阶段 2：设计主机增强通信协议

目标：

- 确定 ESP32 与主机服务的边界

任务：

- 定义音频上传协议
- 定义主机下发命令协议
- 定义 TTS 播放开始/结束协议
- 定义 RESET / STOP / START 等状态控制协议

验收标准：

- 有一份明确的消息协议文档
- ESP32 端具备切换入口

### 阶段 3：实现 Host-Enhanced Mode 最小闭环

目标：

- 主机接管语音重负载能力

任务：

- ESP32 发送 PCM 音频到主机
- 主机 ASR 成功返回文本
- 主机生成动作 JSON
- 主机生成语音回复
- ESP32 播放主机下发的音频并执行动作

验收标准：

- 成功完成一轮“主机增强模式”的语音互动

### 阶段 4：Demo 化整理

目标：

- 为答辩/展示准备最终说法与稳定操作流程

任务：

- 明确展示顺序
- 准备“独立模式”和“增强模式”对比说明
- 收敛调试日志
- 提前准备异常回退策略

验收标准：

- 演示时可以清晰切换或说明两种模式
- 即使主机异常，也可回退到开发板独立方案

---

## 九、近期优先级排序

当前建议按以下优先级推进：

1. 先稳定 `Standalone Mode`
2. 再定义 `Host-Enhanced Mode` 协议
3. 然后接入主机侧流式语音闭环
4. 最后再做更复杂的情绪表达和多轮对话细节

明确不优先做的内容：

- 复杂前端美化
- 过多新外设扩展
- 在板端继续堆更多重计算逻辑

---

## 十、风险与控制

### 10.1 风险

- ESP32 内存与任务调度压力过大
- 主机与开发板之间的网络链路不稳定
- 语音输入与扬声器播放可能互相干扰
- 演示现场网络环境不稳定

### 10.2 控制策略

- 始终保留 Standalone Mode
- 主机模式只接管“最重、最影响体验”的模块
- 所有上层结果都统一为 JSON 动作协议
- 演示时准备主机增强与开发板独立两套流程

---

## 十一、最终展示叙事建议

最终汇报时建议突出以下三点：

1. 系统具备独立运行能力
2. 系统具备主机增强能力
3. 架构允许在资源受限与体验最优之间做工程化平衡

一句话版本：

> 本系统既可以作为独立运行的桌面智能终端，也可以在连接主机后切换为增强模式，以获得更自然的流式语音交互和更稳定的多模态展示效果。

---

## 十二、下一步建议

基于本计划书，下一步最适合立即开展的工作是：

1. 明确 `Host-Enhanced Mode` 的消息协议
2. 抽象 ESP32 端的“语音来源”与“动作执行层”
3. 让主机与板端共享同一套动作 JSON 协议

在这三步完成后，再开始真正接入主机端流式语音服务，会更稳也更不容易返工。
